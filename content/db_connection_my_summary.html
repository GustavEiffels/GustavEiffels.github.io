DB Connection Pool 학습 정리 (나만의 언어로)

2026.04.21 My Study Log

0. 학습 동기

DB를 사용하는 서비스에서 Connection을 매번 새로 만들지 않고 Pool로 관리하는 이유, 그리고 socketTimeout과 Connection Leak이 완전히 다른 문제라는 점을 내 언어로 다시 정리한다.

1. 왜 DB Connection은 "비싼가"?

Connection을 매번 생성하는 것은 세 가지 계층에서 비용을 발생시킨다.

  1. 네트워크 계층 (TCP/TLS): DB 서버와 Application 서버가 다른 머신에 있다면 매번 TCP 3-Way Handshake가 필요하다. TLS까지 적용되면 지연이 더 커진다.
  2. 인증/인가 계층: DB 엔진 레벨에서 ID/PW 검증, 권한(ACL) 조회, 세션 초기화(Charset, Timezone, 트랜잭션 격리 수준 등)가 매번 반복된다.
  3. 자원 점유 계층: 연결 하나당 DB 서버는 독립적인 메모리(스택, 정렬/조인 버퍼)와 스레드/프로세스를 할당한다. 요청마다 할당/회수하면 비용이 누적된다.
"비용은 단순히 '느림'이 아니라, 네트워크 + 인증 + 서버 자원의 복합 비용이다."

2. 그래서 등장한 것: Connection Pool

Connection 객체를 미리 여러 개 만들어 두고 빌려 쓰고 반납하는 방식. 애플리케이션은 0.1ms 미만으로 DB 접근 권한을 획득할 수 있다.

  • Pool: 미리 만들어 둔 Connection들의 집합.
  • Borrow: 요청이 오면 Pool에서 하나 꺼내 사용.
  • Return: 사용이 끝나면 close()로 Pool에 반납.

※ 여기서 close()는 "물리 연결 종료"가 아니라 "Pool에 반납"을 의미한다. 실수하기 쉬운 포인트.

3. Connection의 상태값 (교정된 정리)

Pool이 관리하는 Connection은 다음 상태로 전이한다.

  • Idle: 만들어져 있지만 아무도 쓰지 않는 대기 상태. (= 사용 가능)
  • Active: 애플리케이션이 현재 점유하여 쿼리를 수행 중인 상태.
  • Evicted: 수명이 다하거나 손상되어 폐기된 상태.
📌 자기 교정: 처음엔 "Idle → Ready → Active"로 이해했지만, Ready는 표준 상태 용어가 아니다. 일반적인 Pool(HikariCP 등)은 Idle / Active / Evicted의 3상태로 관리한다.

4. Pool이 만능인가? — 아니다

Pool도 trade-off가 있다.

  • 동시 요청 폭증 시: 여러 애플리케이션 서버가 각자 Pool을 최대치로 열면 DB 서버의 전체 Connection 수가 급증해 DB 자체가 과부하될 수 있다.
  • Idle 연결 유지 비용: 쓰지 않아도 DB 서버의 메모리는 계속 점유된다.
  • 오염된 Connection: 네트워크 단절, DB 재시작 등으로 살아있는 것처럼 보이지만 죽은 연결이 Pool에 남을 수 있다. → max-lifetime, keepalive-time으로 주기적 교체 필요.

5. socketTimeout: 네트워크 응답 대기 상한선

네트워크 장애나 DB 지연으로 Connection이 응답을 무한정 기다리는 것을 막는 설정. 즉, "쿼리를 날린 뒤 응답이 올 때까지 기다리는 최대 시간"이다.

socketTimeout은 네트워크 스트림(송수신)의 지연을 감시한다.

6. 핵심 구분: socketTimeout ≠ Leak 방어

이 둘을 혼동하면 안 된다. 감시 대상이 완전히 다르다.

  • socketTimeout: 네트워크 송수신 지연을 감시. 쿼리 응답이 오지 않을 때 트리거.
  • Leak Detection: Connection의 소유 시간(ownership duration)을 감시. close() 호출이 누락되어 Active 상태로 고착된 경우를 잡아낸다.

예를 들어 쿼리는 끝났지만 코드가 외부 API를 기다리거나 루프를 돌고 있다면 소켓은 조용하다. 이때 socketTimeout은 트리거되지 않고, Connection은 Active로 고착되어 자원이 누수된다.

"쿼리 응답이 안 오는 문제"와 "반납을 안 하는 문제"는 서로 다른 층위의 문제다.

7. socketTimeout이 발동하면 Connection은 어떻게 되나?

  1. Step 1: SocketTimeoutException 발생 및 전파.
  2. Step 2: Pool은 해당 Connection을 신뢰할 수 없다고 판단 → 물리 소켓 강제 폐기(Physical Close).
  3. Step 3: Pool에서 Evicted 처리, 빈자리 제거.
  4. Step 4: 백그라운드 스레드가 minimum-idle 유지를 위해 새 Connection을 생성 시도 → Pool 보충.
📌 자기 교정: 처음엔 "최소 수만큼 다시 만든다"라고 이해했는데, 더 정확히는 손상된 1개를 폐기하고 minimum-idle을 유지하도록 보충하는 것이다. 전체 Pool을 다시 만드는 것이 아니다.

8. 실무 방어 전략 요약

  • leak-detection-threshold (2000~5000ms): 임계 시간 초과 시 스택트레이스로 누수 지점 검거.
  • max-lifetime (약 30분): 잠재적으로 오염된 Connection을 주기적으로 강제 교체.
  • socketTimeout: 네트워크 무한 대기 방지.
  • try-with-resources: 수동 close() 실수 제거. 가장 기본적이면서도 강력한 방어.
단일 설정이 아니라 다층 방어(Defense in Depth)로 접근해야 한다.

Q5. (동작 원리)

Q6. (Trade-off)

Connection Pool이 항상 좋은가? 오히려 DB 서버에 해가 될 수 있는 시나리오를 하나 설명하라.

10. 해설

A1. ① 네트워크(TCP/TLS 핸드셰이크), ② 인증/인가 및 세션 초기화, ③ DB 서버의 메모리·스레드 할당.

A2. Idle(대기 중, 사용 가능) / Active(점유되어 쿼리 수행 중) / Evicted(손상·만료로 폐기됨).

A3. 트리거되지 않는다. socketTimeout은 네트워크 송수신 지연만 감시한다. 쿼리 응답은 이미 받았으므로 소켓은 조용하다. 이 경우는 Connection Leak 문제이며, leak-detection-threshold 또는 try-with-resources로 방어해야 한다.

A4. Connection Leak을 가장 먼저 의심. close() 누락으로 Active 고착된 Connection이 쌓여 Pool이 고갈된 것. leak-detection-threshold를 2000~5000ms로 설정하면 스택트레이스로 누수 지점을 찾아낼 수 있다.

A5. ① SocketTimeoutException 발생·전파 → ② 해당 물리 소켓 강제 폐기 → ③ Pool에서 Evicted 처리 및 빈자리 제거 → ④ 백그라운드 스레드가 minimum-idle 유지를 위해 신규 Connection 생성.

A6. 여러 애플리케이션 서버가 각자 Pool을 최대치(maximum-pool-size)까지 열면 DB 서버 전체 Connection 수가 한계를 넘어 DB가 과부하된다. Pool은 "앱 ↔ DB" 관점에선 효율적이지만, "전체 DB 서버" 관점에선 총량 관리가 필요하다.
