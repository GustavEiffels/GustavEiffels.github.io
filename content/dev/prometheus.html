Prometheus Core System Analysis

1. Prometheus의 정의

📊

Prometheus는 시계열 데이터(Time Series Data) 수집 및 모니터링을 위한 오픈소스 툴입니다. 시스템의 현재 상태를 숫자로 기록하여 시간에 따른 변화를 분석합니다.

메트릭(Metric) 데이터: 시간의 흐름에 따라 변화하는 시스템의 지표입니다. (예: CPU 사용량 80%, 메모리 잔량 4GB 등)

2. 데이터 저장 메커니즘 (Storage)

저장 형식

tdata Format Deep Dive

수집된 데이터는 시계열 최적화 형식으로 로컬 파일 시스템에 저장됩니다.

  • 블록 구조: chunks(데이터), index(색인), meta.json, wal(로그)
  • WAL (Write-Ahead Log): 수집 즉시 로그로 기록하여 장애 시 메모리 데이터 복구 보장
  • 컴팩션(Compaction): 작은 블록들을 병합하여 인덱스 크기 최적화 및 조회 속도 향상
보관 정책

Retention Mechanism

데이터 무결성과 시스템 안정성을 위한 두 가지 보관 기준을 제공합니다.

  • 시간 기준: retention.time 설정 (과거 데이터 자동 폐기)
  • 용량 기준: retention.size 설정 (디스크 Full 장애 방지)
  • 삭제 단위: I/O 최적화를 위해 블록(Block) 단위로 일괄 삭제

3. 빠른 데이터 조회의 비밀 (Indexing)

초당 수십만 개의 데이터가 들어와도 조회가 빠른 이유는 역색인(Inverted Index)과 블록 구조 덕분입니다.

💡 역색인이 일반 DB보다 압도적으로 빠른 이유

일반적인 관계형 DB(RDBMS)에서 WHERE method='GET'을 찾으려면 수백만 줄의 테이블을 위에서 아래로 훑어야 합니다(Full Scan).
반면, Prometheus의 역색인은 마치 책 뒤의 찾아보기(Index)와 같습니다.

[라벨별 색인 지도]
- method="GET": [ID: 1, 5, 10]
- status="200": [ID: 5, 10, 20]
[조회 과정]
"두 리스트에서 겹치는 번호(ID)만 가져와!"
→ ID 5, 10 즉시 추출

데이터가 10억 개가 있어도, Prometheus는 리스트 두 개만 비교하면 되기 때문에 데이터 양에 관계없이 0.001초 만에 응답이 가능합니다.

  • 역색인: 라벨(Label)을 기준으로 데이터 위치를 미리 지도처럼 만들어 둡니다.
  • 청크(Chunk): 데이터를 2시간 단위의 블록으로 묶어 파일로 관리하여 검색 범위를 물리적으로 제한합니다.

4. PromQL (Prometheus Query Language)

수집된 데이터를 필터링하고 연산하는 강력한 쿼리 언어입니다.

PromQL Query 
# 최근 5분간 CPU 사용률 (Idle 제외)
100 - (avg by (instance) (rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100)

* 위 코드는 node_cpu_seconds_total 메트릭의 변화율을 계산하여 전체에서 노는 시간(idle)을 뺀 백분율입니다.

5. Node Exporter

🔌

서버의 하드웨어 및 OS 커널 지표를 수집하여 Prometheus가 읽을 수 있는 HTTP 엔드포인트로 노출합니다.

  • 기본 포트: 9100
  • 주요 수집 메트릭: node_cpu_seconds_total, node_memory_MemAvailable_bytes, node_filesystem_avail_bytes

6. 설치 및 환경 설정 (Installation & Config)

7. Spring Boot 연동 (Micrometer)

Spring Boot 애플리케이션의 내부 메트릭을 노출하기 위해 Micrometer와 Actuator를 활용합니다.

// build.gradle
implementation 'io.micrometer:micrometer-registry-prometheus'
// application.yml
management.endpoints.web.exposure.include: 'prometheus'

위 설정 후 /actuator/prometheus 경로를 통해 JVM 상태, 힙 메모리, 쓰레드 정보 등이 노출됩니다.

8. 엔지니어링 고려 사항: Pull vs Push

Prometheus의 Pull 기반 아키텍처는 모니터링 시스템의 복잡도를 낮추고 안정성을 높이는 데 핵심적인 역할을 합니다.

특징 내용
중앙 집중 부하 조절 수집 서버가 주도권을 가지므로 대상 서버에 부하를 주지 않음
서비스 가동성 확인 Pull 실패 시 즉시 'Up/Down' 상태 판별 가능

9. TSDB 데이터 구조 심층 분석 (Block Storage)

Prometheus TSDB 블록은 독립적인 DB 역할을 수행하는 물리적 단위입니다.

📁 chunks/

실제 샘플 데이터 저장소. 시계열 데이터가 압축되어 불변 상태로 보관됩니다.

🔍 index

역색인 정보 포함. 라벨을 통해 시리즈 ID와 청크 위치를 고속 검색합니다.

📝 wal/

Write-Ahead Log. 메모리 데이터가 블록화되기 전까지 장애 복구를 위해 선기록됩니다.

📄 meta.json

시간 범위, 샘플 수, 컴팩션 레벨 등 블록의 상태 정보를 기술합니다.

10. 컴팩션 (Compaction) 및 데이터 수명 주기

10.1 컴팩션 메커니즘

데이터가 축적됨에 따라 발생하는 성능 저하와 디스크 낭비를 해결하기 위해 컴팩션을 수행합니다.

병합 및 최적화 효과:
  • 쿼리 최적화: 수집된 여러 블록을 하나로 합쳐 탐색해야 할 인덱스 파일 수를 최소화합니다.
  • 중복 제거: 여러 블록에 걸쳐 나타나는 동일한 라벨 정보를 통합하여 인덱스 크기 줄입니다.
  • 오버랩 해결: 데이터 수집 지연으로 발생한 시간 범위 중복을 논리적으로 정렬합니다.

10.2 데이터 저장 및 로드 흐름 (Full Lifecycle)

STEP 1

수집 및 선기록 (Scrape & WAL)

메트릭을 읽어오는 즉시 WAL에 기록하여 전원 차단 등에 대비한 영구 저장성을 확보합니다.

STEP 2

메모리 적재 (Head Block)

실시간 조회를 위해 메모리 공간인 Head Block에 데이터를 유지합니다. 이 시점부터 쿼리가 가능합니다.

STEP 3

블록화 (Checkpointing)

약 2시간 분량의 데이터가 모이면 메모리 데이터를 디스크 상의 불변 블록(Block)으로 변환합니다.

STEP 4

조회 및 로드 (Query Execution)

사용자 쿼리 발생 시, Index 파일을 메모리 매핑(Mmap)하여 특정 시리즈를 찾고, 필요한 Chunks 데이터만 로드하여 반환합니다.

11. 역색인(Inverted Index)의 내부 동작 및 조회 과정

Prometheus가 대규모 데이터에서 특정 시계열을 찾는 4단계 내부 메커니즘입니다.

Step 1. Posting List 획득

각 라벨(method="POST", status="500")에 대응하는 정렬된 시리즈 ID 리스트를 메모리로 가져옵니다.

List A: [10, 25, 42, 100] / List B: [25, 42, 88, 100]
Step 2. 고속 교집합 연산 (Intersection)

두 리스트가 정렬되어 있으므로, 단 한 번의 스캔으로 겹치는 ID를 찾아냅니다. (컴퓨터 공학의 Merge 조인 방식)

Result: Series IDs [25, 100]
Step 3. 시리즈 위치 참조

인덱스 파일 내 Series Section을 참조하여, 해당 ID가 어느 chunks/ 파일의 몇 번째 줄(Offset)에 있는지 파악합니다.

Step 4. Chunk 로드 및 반환

파일 전체를 읽지 않고, 필요한 오프셋 데이터만 디스크에서 메모리로 올려 사용자에게 즉시 전달합니다.

💡 핵심 인사이트:

RDBMS는 조건에 맞는 행을 찾기 위해 테이블을 뒤지지만, Prometheus는 숫자 리스트 간의 비교 연산으로 조회를 끝냅니다. 이것이 바로 Prometheus가 '시계열 데이터의 왕'으로 불리는 이유입니다.

12. 실시간 데이터 처리의 핵심: 2시간의 마법 (Head Block & WAL)

Prometheus는 최신 데이터의 무결성과 성능을 보장하기 위해 메모리(Head Block)와 로그(WAL)를 병행 운용합니다.

🧠 메모리 (Head Block)

실시간 수집 데이터를 보관하며 즉시 쿼리가 가능하도록 지원합니다. (수집 후 ~2시간)

🛡️ 보험 (WAL)

서버 장애 시 메모리 데이터를 복구하기 위해 디스크에 한 줄씩 기록하는 로그입니다.

🔄 2시간 이후의 데이터 변화 (Promotion)

데이터가 2시간 분량이 쌓이면 Checkpointing이 발생합니다. 메모리의 데이터를 내려받아 정식 Block을 생성하고, 이제는 불필요해진 이전의 WAL 기록들을 삭제하여 디스크를 효율적으로 관리합니다.

이 구조 덕분에 Prometheus는 '실시간 조회 성능'과 '장애 시 데이터 복구'라는 두 마리 토끼를 동시에 잡을 수 있습니다.
⚙️ Head Block 주기 조절 (Expert Only)

기본 2시간 주기는 --storage.tsdb.min-block-duration 플래그로 조절 가능합니다.
- 주기 단축: 메모리 사용량 감소, 디스크 I/O 증가
- 주기 연장: 메모리 사용량 급증 (비권장)

13. Series ID의 역할과 데이터 매핑 매커니즘

라벨 검색 결과로 얻은 Series ID(정수)가 실제 데이터로 변환되는 기술적 과정입니다.

왜 숫자를 쓰는가?

  • CPU 친화적: 문자열 비교보다 정수 비교가 수만 배 빠름
  • 공간 효율: 라벨 중복 저장을 막아 디스크 사용량 감소

매핑 방식

  • Index: ID → Chunk 파일 오프셋 지도
  • Chunk: 실제 압축된 시계열 데이터 저장소

📚 도서관으로 비유하는 데이터 로드

1. 검색: "자바" 책을 도서 검색대에서 조회 (역색인)
2. 기호 획득: 결과로 IT-25라는 청구 기호 획득 (Series ID)
3. 지도 확인: IT-25는 "3층 5번 선반"에 있음을 확인 (Index 지도 참조)
4. 책 대출: 실제 선반으로 가서 해당 책을 꺼내어 읽음 (Chunk Data 로드)
