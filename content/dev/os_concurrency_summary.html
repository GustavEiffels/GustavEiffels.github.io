OS 프로세스 제어와 동시성

기준 시점: 2026.04.12 OS Engineering · Concurrency

1. 개요: 시스템 엔지니어링

프로세스 제어부터 동시성 해결까지, 현대 소프트웨어의 근간이 되는 OS의 메커니즘을 이해한다.

2. OS Core: 프로세스, 스레드, IPC

시스템을 구성하는 세 가지 핵심 실행 단위와 소통 방식이다.

🏰 프로세스

독립된 인스턴스

  • 메모리 격리 (Heap/Stack)
  • PCB 관리
  • 중량급 컨텍스트 스위칭

👷 스레드

최소 실행 흐름

  • Heap/Data 공유
  • 독립적 Stack/Register
  • 경량급 실행 전환

⚓ IPC

프로세스 간 소통

  • 커널 중계 시스템
  • FD 상속 기반 통로
  • 동기화/Blocking 내장

3. 프로세스 관계와 소통의 본질

부모와 자식 프로세스는 단순한 위계 관계를 넘어 자원 관리의 책임을 공유한다.

개념 메커니즘 핵심 역할
부모-자식 Fork() & Wait() 부모는 자식의 리소스(PCB 등) 수거 책임
FD 상속 소유권 이전 격리된 환경에서도 스트림 통로 공유 가능
표준 스트림 0(STDIN), 1(STDOUT), 2(STDERR) OS가 부여한 기본 3대 통로
📌 경고: 부모가 wait()를 통해 종료 신호를 수거하지 않으면 자식은 좀비(Zombie) 프로세스가 리소스 회수가 불가능하다.

4. 블로킹(Blocking)과 교착 상태(Hang)

OS 커널 버퍼(약 64KB)의 한계로 인해 발생하는 병목 현상을 시각화한다.

🛑 Write Blocking

현상: 커널 버퍼 FULL

원인: 자식은 계속 쓰는데 부모가 읽지 않을 때

결과: 자식 프로세스 무한 대기

⏳ Read Blocking

현상: 커널 버퍼 EMPTY

원인: 부모가 읽으려는데 데이터가 없을 때

결과: 부모 스레드 중단(Wait)

"부모가 waitFor()로 자식을 기다리면서 동시에 스트림을 읽지 않으면, 64KB 버퍼가 차는 순간 시스템은 영원히 멈춘다."

5. Java Race Condition

여러 스레드가 공유 자원에 동시 접근할 때 발생하는 데이터 오염 현상

count++; // Read-Modify-Write의 비원자적 특성

✅ 해결 A: Lock

synchronized나 ReentrantLock으로 상호 배제(Mutex) 보장.

✅ 해결 B: Atomic

AtomicInteger 등 CAS(Compare-And-Swap) 연산 활용.

6. 실무 전략: Stream Gobbler

자바에서 외부 프로세스를 안전하게 제어하기 위한 표준 전략

// 별도 스레드에서 실시간 스트림 드레인(Drain)

new Thread(() -> {

process.getInputStream().transferTo(System.out);

}).start();

int exitCode = process.waitFor(); // ✅ 안전하게 종료 대기

7. 핵심 마스터 퀴즈 (Self Quiz)

Q1. 커널 스트림 버퍼의 임계치와 초과 시 현상은?

OS가 관리하는 버퍼의 일반적인 크기와, 이 한계에 도달했을 때 자식 프로세스에 일어나는 일은?

💡 해설: 약 64KB. 버퍼가 꽉 차면 자식 프로세스는 Write Blocking 상태가 되어 더 이상 실행되지 못하고 멈춘다.

Q2. FD 상속의 보안적 의미는?

왜 부모-자식 관계가 아닌 제3의 프로세스는 자식의 STDOUT을 읽을 수 없는가?

💡 해설: OS는 FD(File Descriptor) 테이블 소유권을 철저히 격리한다. 상속을 통해 통로를 공유받지 않은 프로세스는 물리적으로 해당 버퍼에 접근할 수 없다.

Q3. 좀비 vs 고아 프로세스

자식이 종료되었으나 부모가 wait()를 호출하지 않아 PCB가 남은 상태는 무엇인가?

💡 해설: 좀비(Zombie) 프로세스. (참고: 부모가 먼저 죽어 자식만 남은 경우는 고아(Orphan) 프로세스라고 하며, 보통 init 프로세스가 수거한다.)

8. 최종 정리 (Final Summary)

64KB 버퍼의 물리적 한계를 인지하고 비동기 소비 전략(Stream Gobbler)을 수립하고 공유 자원 접근 시 원자성을 보장하는 것이 견고한 시스템을 만드는 방법이다.
