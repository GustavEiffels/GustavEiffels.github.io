DB Connection이 "비싼(Expensive)" 이유: 기술적 메커니즘 분석

2026.04.20 Database

1. 개요: 왜 매번 연결하지 않는가?

애플리케이션 개발 시 데이터베이스 연결을 필요할 때마다 만들고 끊는 것은 성능에 치명적입니다. 흔히 "커넥션은 비싸다"고 표현하는데, 이는 물리적인 네트워크 계층부터 애플리케이션 프로토콜 계층, 그리고 DB 서버의 OS 자원 계층에 이르기까지 복합적인 비용이 발생하기 때문입니다.

2. 비용 발생의 3대 핵심 원인

① 네트워크 계층: TCP/IP 3-Way Handshake

DB 서버와 애플리케이션 서버가 통신을 시작하려면 TCP 연결이 확립되어야 합니다.

  • Handshake: SYN -> SYN/ACK -> ACK 과정에서 최소 3번의 네트워크 왕복(RTT)이 발생합니다.
  • TLS/SSL 보안: 최근 보안 강화로 인해 암호화 연결을 사용할 경우 추가적인 핸드쉐이크가 발생하여 지연 시간(Latency)이 2~3배 증가합니다.

② 프로토콜 및 인증 계층: 보안 및 환경 설정

네트워크가 연결된 후에도 DB 엔진 레벨의 검증 과정이 남아있습니다.

  • 인증(Authentication): 사용자의 ID/PW 확인 및 접근 권한(ACL) 조회.
  • 세션 초기화: 언어(Charset), 타임존, 트랜잭션 격리 수준(Isolation Level) 설정 등 논리적인 세션 생성 비용이 발생합니다.

③ 자원 점유 계층: 서버 측 메모리 및 CPU 할당

연결된 사용자마다 DB 서버는 물리적 자원을 점유합니다.

  • 메모리 점유: 각 연결(Connection)마다 스택 영역과 정렬/조인 등을 위한 전용 버퍼 메모리를 할당합니다.
  • 컨텍스트 스위칭: DB 서버가 수많은 커넥션을 스레드/프로세스로 관리하면서 발생하는 CPU 오버헤드는 실제 쿼리 수행 성능을 저하시킵니다.

3. 해결책: Connection Pool (HikariCP)

이러한 고비용 작업을 매번 수행하지 않기 위해 **Connection Pooling** 기술을 사용합니다.

"미리 연결을 만들어 풀(Pool)에 보관하고, 필요할 때 빌려 쓰고 반납하는 방식"

이를 통해 애플리케이션은 0.1ms 미만의 빠른 시간 안에 DB 접근 권한을 획득할 수 있게 됩니다.

4. 엔지니어의 핵심 요약

  1. 네트워크: TCP 3-Way Handshake + SSL 지연 시간 발생.
  2. 인증: DB 엔진 레벨의 권한 검사 및 세션 초기화 작업 필요.
  3. 서버 자원: 연결당 고정 메모리 점유 및 스레드 관리 오버헤드 발생.
