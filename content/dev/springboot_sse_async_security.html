SSE & ASYNC Dispatch - Spring Security 인가 심층 분석

Spring Security SSE Async 2026-04-23

SSE(Server-Sent Events) 환경에서 Spring Security의 ASYNC Dispatch 인가 처리 흐름을 분석하고, 실무 해결 코드까지 정리한 리서치 노트입니다.

1. SSE(Server-Sent Events) 메커니즘

클라이언트가 1회 연결 요청 후, 서버가 일방적으로 데이터를 푸시하는 Persistent Connection 방식입니다.

비유: '구독형 신문'
한번 구독 신청(REQUEST)을 하면, 서버는 매일 신문을 배달(ASYNC Dispatch)합니다. 배달원이 문 앞에서 매번 신분증을 요구하면 배달이 지연되는 것과 같은 원리입니다.

2. Technical Deep Dive (기술적 인과관계)

🧵 Thread & ThreadLocal

Spring Security는 인증 정보를 ThreadLocal에 저장합니다. 비동기 스레드나 ASYNC Dispatch 과정에서 스레드가 교체되면 기존 보안 컨텍스트가 유실되어 미인증 사용자로 간주됩니다.

📄 File Descriptor (FD)

SSE는 소켓(FD)을 점유하여 스트리밍을 이미 시작한 상태(Committed)입니다. 이 상황에서 보안 에러로 인해 리다이렉트를 시도하면 충돌 에러가 발생합니다.

3. 전체 동작 흐름도

SSE Dispatch Flow 
클라이언트                    Tomcat/Spring
    │                              │
    │── GET /status-stream ───────>│  ← [1단계] 클라이언트 단발성 요청
    │   (쿠키/세션 포함)            │  (REQUEST dispatch, 인가 통과)
    │                              │
    │<── 연결 유지 ────────────────│  ← [2단계] 응답 헤더 전송 (Committed)
    │                              │
    │          [이벤트 발생]        │
    │                              │──> [3단계] Tomcat 내부: ASYNC dispatch 생성
    │                              │    (서버 내부적인 재시작 흐름)
    │                              │
    │                              │    Spring Security 재인가 시도
    │                              │    → SecurityContext 없음
    │                              │    → Access Denied ← [4단계] 에러 발생!
    │                              │
    │<── 데이터 전송 (차단됨)        │

4. 최종 정리: 엔지니어링 관점의 이해

  • ✅ 등록은 단 한 번: 클라이언트는 서버에게 처음에만 등록 요청을 보내고, 이후엔 가만히 서있기만 합니다.
  • ✅ 내부적 재작동: 서버가 클라이언트 정보를 받아 확인한 후, 내부적으로 비동기 처리를 위한 ASYNC dispatch를 생성합니다.
  • ✅ 데이터 투척: 서버는 일이 생기면 등록된 클라이언트 정보를 꺼내와서 ASYNC dispatch를 통해 데이터를 던져줍니다.
  • ⚠️ 보안 필터의 오해: 이때 Spring Security가 서버 내부의 움직임을 새로운 외부 요청으로 착각하여 인가를 다시 검사하다가 정보를 못 찾아 에러를 냅니다.

5. 해결 방법 (Java 실무 코드)

1
SecurityConfig.java 설정 추가
SecurityConfig.java 
// SecurityConfig.java
http.authorizeHttpRequests(authorize -> {
    authorize
        // 💡 핵심: "서버 내부의 ASYNC 흐름은 이미 검증된 것이니 통과시켜라"
        .dispatcherTypeMatchers(DispatcherType.ASYNC).permitAll()
        .requestMatchers("/api/sse/**").authenticated()
        .anyRequest().authenticated();
});

핵심 포인트 DispatcherType.ASYNC 를 permitAll() 처리하면, 서버 내부에서 발생하는 ASYNC Dispatch는 Spring Security 인가 체크를 건너뜁니다. 최초 REQUEST 단계에서 이미 인가가 완료되었기 때문에 보안상 안전합니다.
